**Responsible AI Assistance & Validation Report**

## Section 1: Introduction

### Project Context
**ExpenseFlow** is a production-grade, multi-currency shared expense management system featuring dynamic membership timelines, mathematical debt minimization, and an interactive CSV diagnostics wizard. To build this application within the project timeline, AI assistance was integrated throughout the planning, scaffolding, implementation, and documentation phases.

### Document Purpose
This report serves as the **AI Usage Log** for the project. In accordance with professional engineering standards, it documents:
1. The AI tools employed and their specific utility.
2. The key prompts used to guide AI code generation.
3. Five concrete examples where the AI produced incorrect, suboptimal, or policy-violating code, explaining how these mistakes were detected and corrected.
4. The manual and automated verification processes used to validate the final system.

### Development Philosophy
Our development approach utilized AI as an **interactive code collaborator** rather than an automated replacement for human engineering. The core tenets of this philosophy were:
* **Manual Code Ownership**: Every single line of code generated by an AI assistant was read, analyzed, and approved by a developer before being committed.
* **Requirements Over Algorithms**: AI suggestions were compared against the project requirements. If an AI tool recommended a design that conflicted with the project guidelines, the AI suggestion was rejected.
* **Verifiable Code Execution**: AI-generated code was subjected to rigorous manual verification, unit testing, integration tests, and database schema validation to ensure correctness.
* **Product Thinking Primacy**: User experience, auditability, and data safety decisions were driven by human product design rather than default code templates.

## Section 4: AI Mistakes and Corrections

This section documents five concrete examples where the AI produced incorrect, suboptimal, or policy-violating code, explaining how these issues were detected and corrected.

---

### Example 1: Suggesting MongoDB instead of a Relational Database

* **Problem**: The AI suggested using MongoDB as the primary database, providing code templates for Mongoose models and explaining that a document-based design is ideal for storing dynamic participant arrays.
* **Why it was wrong**: This suggestion violated the core project requirements, which specified a relational PostgreSQL database. Using a non-relational database like MongoDB would make it harder to enforce foreign key constraints, cascading deletes, and strict transactional rollbacks. These features are critical for ensuring financial ledger consistency.
* **How it was detected**: Manual requirements check: The developer reviewed the generated models and compared them against the project constraints.
* **Final Correction**: The database design was migrated to **PostgreSQL** using **Prisma ORM**. The relationships between models (`User`, `Group`, `GroupMember`, `Expense`, and `ExpenseParticipant`) were defined using foreign keys and cascading updates.

```prisma
// Corrected Relational Setup in schema.prisma
model ExpenseParticipant {
  id                    String      @id @default(uuid())
  expenseId             String
  expense               Expense     @relation(fields: [expenseId], references: [id], onDelete: Cascade)
  memberId              String
  member                GroupMember @relation(fields: [memberId], references: [id], onDelete: Cascade)
  shareValue            Float       @default(0)
  calculatedShare       Float
  calculatedShareInBase Float
}
```

### Example 2: Storing Debt Settlements as Expense Records

* **Problem**: The AI suggested storing peer-to-peer repayments (settlements) in the `Expense` table, using negative amounts to differentiate them from normal expenses.
* **Why it was wrong**: Storing settlements in the `Expense` table with negative amounts introduces two major issues:
  1. It distorts group spending analytics: repayments would register as negative expenses, incorrectly reducing the group's total spending metric.
  2. It complicates calculations: the balance calculation algorithm would need to filter out negative expenses to avoid double-counting adjustments.
* **How it was detected**: During manual testing of the balance calculation engine, the developer noticed that group spending summaries were incorrect and balances drifted after logging repayments.
* **Final Correction**: Created a separate `Settlement` table in the database and updated the balance calculation logic to process expenses and settlements independently.

```typescript
// Corrected Dynamic Balance Calculation Logic in Node.js
const totalPaid = member.paidExpenses.reduce((sum, e) => sum + e.amountInBase, 0);
const totalOwed = member.shares.reduce((sum, s) => sum + s.calculatedShareInBase, 0);
const settlementsPaid = member.paidSettlements.reduce((sum, s) => sum + s.amountInBase, 0);
const settlementsReceived = member.receivedSettlements.reduce((sum, s) => sum + s.amountInBase, 0);

// Net Balance formula: Credits (paid) minus Debits (owed) adjusted by settlements
const netBalance = (totalPaid - totalOwed) - settlementsPaid + settlementsReceived;
```

### Example 3: Automatically Deleting Duplicate Expenses during CSV Ingestion

* **Problem**: The AI suggested automatically deleting or skipping duplicate transactions during CSV parsing, without notifying the user.
* **Why it was wrong**: Silently deleting data based on assumptions can cause user frustration. In group settings, identical transactions can happen on the same day (e.g., buying two separate rounds of coffee at the same shop for the same price). Automatically dropping duplicates would lead to missing expenses.
* **How it was detected**: The developer reviewed the code and noticed that duplicate check rows were dropped during parsing:
  ```typescript
  // Suboptimal AI Code
  if (isDuplicate(row)) continue; // Silently deleted
  ```
* **Final Correction**: The import engine was updated to flag duplicates as warnings in the database (`ImportIssue` table). The frontend was updated to display these warnings, giving users the choice to skip the duplicate or import it anyway.

```typescript
// Corrected Check in backend/src/utils/anomaly.ts
if (duplicateExact) {
  anomalies.push({
    issueType: 'DUPLICATE',
    severity: 'WARNING',
    description: `An identical expense '${rowData.title}' of ${rowData.amount} exists in the ledger`,
    suggestedAction: 'Reject duplicate import, or import as separate transaction'
  });
}
```

### Example 4: Ignoring Membership Timelines in Split Calculations

* **Problem**: The AI wrote a split calculator that divided costs among all group members in the database, ignoring their active membership dates.
* **Why it was wrong**: Enforcing membership timelines is a core project requirement. Dividing costs among members who had not yet joined or had already left the group causes incorrect billing.
* **How it was detected**: Manual verification: The developer set up a test group where a member left in March and another joined in April. The AI split an April expense with the inactive member, violating the timeline rule.
* **Final Correction**: The calculation logic in the imports controller was updated to check membership timelines. It filters the group roster, ensuring splits only apply to members who were active on the date of the expense.

### Example 5: Overwriting Original Currency Details during Conversion

* **Problem**: The AI suggested converting multi-currency expenses immediately during import and storing only the converted value in the database, discarding the original amount and currency details.
* **Why it was wrong**: Overwriting the original values destroys the audit trail. Users need to see original transaction details (e.g., the currency and amount listed on the receipt) to verify that conversions and splits were calculated correctly.
* **How it was detected**: The developer reviewed the database schema and noticed that the `Expense` table only had a single `amount` field, which was populated with the converted value.
* **Final Correction**: The `Expense` model was updated to store all conversion details: `amount` (original amount), `currency` (original currency), `amountInBase` (converted amount in base currency), and the exchange rate was stored in the split model `ExpenseParticipant`.

```prisma
// Corrected Database Columns in schema.prisma
model Expense {
  id           String      @id @default(uuid())
  amount       Float       // Original amount (e.g., 120.00)
  currency     String      @default("USD") // Original currency (e.g., "USD")
  amountInBase Float       // Converted amount (e.g., 100.00 EUR)
}
```

## Section 5: Validation Process

To ensure the application was reliable and met all requirements, a validation process was implemented:

### 1. Manual Code Reviews
Every file and model generated by the AI was reviewed line-by-line. This helped find logical bugs, security gaps (such as missing JWT validations), and clean up unnecessary code comments.

### 2. CSV Import Testing
Test files were uploaded to verify that the CSV validation engine works correctly. This included files with:
* Malformed rows (too many/few columns) to test column mismatch handling.
* Empty fields to check required field validation.
* Zero or negative amounts to test range validations.
* Emails not in the group roster to test the auto-invite workflow.
* Future dates and unsupported currencies to test warning fallbacks.

### 3. Database Inspections
Prisma Studio was used to inspect the database after import and transaction runs. This helped verify that:
* Foreign keys and relationships mapped correctly.
* Cascade delete rules worked as expected.
* Exchange rates were stored correctly.
* Audit logs were created for administrative actions.

### 4. Interactive Calculations Verification
The math engines (debt solver and split calculators) were tested by comparing their results against manual spreadsheets. This verified that:
* The debt solver correctly minimized transfer routes.
* Rounding remainders (cents) were handled correctly without database inconsistencies.

## Section 6: Human Decisions

Several design and product decisions were made independently of AI recommendations to ensure the application met project standards:

1. **Relational Database Design**: Enforced PostgreSQL as the primary database database engine to guarantee ACID transaction support and foreign key constraints, rejecting AI suggestions for NoSQL databases.
2. **Explicit Verification Workflow**: Decided to build an interactive CSV diagnostic editor that lets users review and correct anomalies inline, rather than automatically correcting or discarding rows.
3. **Dedicated Settlement Table**: Chose to isolate peer-to-peer repayments in a separate database model to keep expense analytics clean and accurate.
4. **Timeline Validation Engine**: Built date comparison logic to enforce active membership timelines on splits, preventing inactive members from being billed.

## Section 7: Lessons Learned

Integrating AI into the development of ExpenseFlow highlighted several key engineering principles:

* **AI Accelerates Velocity, Not Judgment**: AI tools are helpful for generating boilerplate code and templates, but they cannot make product decisions or ensure database integrity. The developer is responsible for the overall system architecture.
* **Requirements Take Precedence**: AI recommendations should always be validated against the project requirements. If a tool suggests a design that violates these guidelines, it must be rejected.
* **Test the Edge Cases**: AI systems tend to write code that assumes clean inputs. Manual verification and edge case testing are required to ensure code handles invalid inputs and logical conflicts safely.
* **Prioritize Transparency**: In financial applications, data safety and auditability are critical. Features that highlight issues and request user approval build trust and prevent data corruption.
* **Understand the Code**: Every line of code generated by an AI assistant should be understood by the developer. This is essential for debugging and explaining system behavior in technical interviews.
